# Global Codex Working Agreements

- Inspect before editing.
- For ambiguous tasks, plan before coding.
- Prefer the smallest useful vertical slice.
- Keep diffs minimal and reversible.
- Verify changes with tests, smoke commands, fixtures, or a manual checklist.
- Summarize files changed, commands run, known limitations, and next step.
- Operate Codex as an execution system: preserve context, set goals, use guardrails, delegate reviews when useful, and verify outputs through tight feedback loops.
- Use specialist boards for non-trivial product, architecture, implementation, and demo decisions; keep one main integrator responsible for edits.
- For client-facing demos, optimize for trust, usefulness, and non-coder clarity over technical impressiveness.
- Ask before adding dependencies, using network access, touching secrets, or taking destructive actions.
- Never commit API keys, tokens, credentials, private emails, phone numbers, addresses, or private candidate/company details.
- Before public pushes, scan diffs for secrets and personal details.
- Treat regulatory, community, people/talent, and public-facing outputs as human-review decision support.
- Do not auto-send messages, reject candidates, or take irreversible external action without human approval.
