# Hackathon Day Checklist

## Before challenge starts

- Confirm environment works.
- Inspect repo.
- Read provided instructions.
- Identify app/test commands.
- Start a notes file.

## Operational context

- Follow event rules.
- Be present and curious.
- Do not record or capture private materials unless explicitly allowed.
- Ask thoughtful questions only if natural.
- Look for operational bottlenecks, reliability themes, handoff friction, and places internal tools could help.

## Problem show

- Listen for the owner, workflow, pain, constraints, and desired outcome.
- Write down trusted inputs and required outputs.
- Identify what would be useful by demo time.
- Ask what not to build.
- Ask what happens if the tool is wrong.

## During problem discovery

- Ask who the operator is.
- Ask what workflow is slow today.
- Ask what data is trusted.
- Ask what output would be useful by tomorrow.
- Ask what not to build.
- Ask what makes demo compelling.

## Paired execution

- Align with partner in first 5 minutes.
- Decide owner roles.
- Set sync checkpoints.
- Keep one coherent artifact.
- Integrate early.
- Avoid parallel disconnected work.

## Build strategy

- Ship smallest useful slice first.
- Keep tests/checks running.
- Avoid broad rewrites.
- Prefer boring tech.
- Use Codex for repo analysis, implementation, debugging, and review.
- Use my judgment for product scope.

## Demo

- Pain
- Messy input
- Tool
- Output
- Validation
- Limitations
- Next iteration

## After demo / informal event

- Be curious.
- Ask what surprised them about the organization.
- Ask what makes FDEs successful.
- Ask what real operational workflows are hardest.
- Do not overpitch.
- Be normal, energetic, mission-aligned.

## Project recap / follow-up

Be ready to explain:

- what we built
- why we scoped it that way
- how problem-owner feedback changed the build
- how Codex helped
- where human judgment mattered
- what productionizing would require
