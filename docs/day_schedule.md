# Event Day Operating Blocks

Use this as a generic onsite hackathon operating plan. Keep private company names, exact locations, named people, and event-specific logistics out of this public repo.

| Block | Segment | What Is Being Evaluated | What To Do | Artifact |
|-------|---------|--------------------------|------------|----------|
| Arrival | Check-in and operations context | Presence, curiosity, operational awareness | Arrive ready, follow device rules, observe real workflow constraints | Mental notes only |
| Briefing | Schedule, rules, and technical context | Rule comprehension, judgment, listening | Capture constraints, allowed tools, expected deliverables, demo format | Brief rules note |
| Discovery | Problem-owner conversation | User discovery and scoping | Understand workflow, pain, trusted inputs, desired output, risk, non-goals | Discovery notes |
| Spec | Agent/tool spec document | Ability to convert ambiguity into a buildable plan | Write a concise spec for evaluators and owners | One-page agent/tool spec |
| Build 1 | First implementation block | Execution, follow-up questions, early validation | Build the smallest useful vertical slice and ask owners for feedback | Working MVP path |
| Interruption | Interview, lunch, or owner meeting | Communication and context discipline | Preserve context before leaving; resume from build state | Build-state note |
| Build 2 | Iteration block | Tradeoffs, robustness, owner feedback | Finish useful artifact, validation, demo path, fallback | Demoable artifact |
| Demo | Final presentation | Usefulness, trust, non-coder clarity | Demo pain -> messy input -> tool -> output -> trust -> next | Timeboxed demo |
| Follow-up | Informal or next-day conversations | Reflection, role fit, production judgment | Explain what you built, why, tradeoffs, and next steps | Project recap answers |

## Critical Timing Pattern

- User discovery is short. Use it to learn the real workflow, not to pitch prematurely.
- The agent/tool spec block is short. Produce a practical one-page build plan.
- Interviews or meetings may interrupt the build. Preserve context before leaving.
- Evaluators may ask casual questions during build. Keep a current 30-second answer ready.
- Demo timing may vary. Prepare 30-second, 3-minute, and 5-minute variants.

## Discovery-To-Spec Operating Loop

1. Capture owner, user, workflow, pain, trusted inputs, required output, constraints, and risk.
2. Ask what would be useful by demo time and what not to build.
3. Choose one artifact type.
4. Define the smallest useful version and explicit non-goals.
5. Write the spec.
6. Confirm the spec with the problem owner if possible.
